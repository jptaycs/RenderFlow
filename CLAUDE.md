# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

RenderFlow is an AI video production **orchestration platform**: topic or script in → narrated, rendered MP4 out. It coordinates external AI providers (LLM, image, TTS, avatar) behind swappable interfaces — it is not a model itself. Target: long-form YouTube documentaries (20–30 min), currently at the Week-1 walking-skeleton stage producing short videos end-to-end.

`AGENTS.md` points here; this file is the single source of truth — keep it honest as things change.

**Documentation rule: every change updates this file, in the same session.** If a session changes behavior, adds/renames a file or env var, installs a dependency, switches a provider or voice, learns a provider quirk the hard way, or verifies something against a live service — record it here (Current state, Commands, Configuration, Provider notes, Layout, whichever fits) before finishing. Anything not written down here is knowledge the next session doesn't have.

## Current state

Working today, verified against live services:

- Synchronous CLI pipeline: scene plan → images → voice → FFmpeg 1080p MP4
- Providers: Claude (LLM, untested live — no API key yet), Pollinations + Flux-Replicate (image), Kokoro (**default TTS** — local, natural prosody) + Piper + ElevenLabs (TTS)
- Resumable asset generation (completed assets skip on re-run), per-asset cost tracking
- Talking-avatar scene contract with four providers: `wav2lip-local` (**default in practice** — free unlimited local lip sync + camera push-in, verified live, ~real-time on CPU), `ffmpeg-still` placeholder (static image + voice; no lip sync), `sadtalker-replicate` (paid lip sync + real head motion via Replicate; written, unverified — needs a funded `REPLICATE_API_TOKEN`), and `memo-hf` (free HF Space, demo-only: 4s audio cap)
- Piper narration pacing: `RENDERFLOW_TTS_LENGTH_SCALE` + sentence-pause insertion (see Provider notes)
- Fast visual pacing: scripts split into ~5-second scenes (clause-level splits for long sentences); image prompts are written as realistic photo captions with anti-AI-look negative prompts. Composition rule: **no visible human face at all** (strengthened 2026-07 from "no lone close-up," which still allowed faces in wider shots, per direct client feedback) — if a person appears, show them from behind/silhouette/distance/hands-only, enforced in both LLM prompts and the local prompt builder + negative prompts. `_local_image_prompt` leads with the constraint *before* the narration content (diffusion models weight earlier tokens more heavily), but prompt wording alone is **still not reliable** on abstract narration with no concrete noun ("the difference is not fancy technology") — Pollinations/Flux folds `negative_prompt` into plain text rather than true CFG-negative conditioning, so it's a suggestion the model can ignore.
- **Face-detection retry** (`pipeline/facecheck.py`): because prompt wording alone isn't reliable enough, scene-image and thumbnail-background generation (`assets._generate_face_free`, NOT the avatar portrait or thumbnail reaction face — those are supposed to have a face) now check their own output with OpenCV's YuNet DNN face detector and regenerate (fresh random seed each call) up to `MAX_FACE_RETRIES = 3` times if a face covers ≥12% of the frame's shorter side; if still failing after 3 tries, keeps the last result and logs a warning rather than blocking the scene. Verified live: converts some previously-100%-failing abstract lines to face-free within 1-2 retries, but a genuinely stubborn line can still fail all 3 attempts — an honest mitigation, not a guarantee. **Do not use `cv2.CascadeClassifier`/Haar cascades** — this opencv-python build (verified 5.0.0) removed that API entirely (`AttributeError`, not just missing data files). The YuNet ONNX model (~230 KB) auto-downloads once to `.cv2_models/` (gitignored) — small enough to skip a dedicated setup script, unlike wav2lip/kokoro.
- Local web dashboard (`renderflow/api.py` + `web/index.html`): FastAPI serves project state, live scene thumbnails, and launches `make_video.py` runs as subprocesses (create / resume / regenerate-scene). Finished renders are download-only by design (no inline player — client preference); the download card shows the project thumbnail. The download link's suggested filename is the project title (`downloadFilename()` in `web/index.html`, sanitized for filesystem-illegal characters), not the on-disk `final.mp4` — client-side only, via the anchor's `download="..."` attribute; the file on disk is untouched. Verified live end-to-end. Dev stand-in for the Week-2 worker queue — pipeline never runs inside a request handler.
- Project thumbnail: a **generated clickbait image + host reaction badge**, not a scene crop. `assets.generate_thumbnail()` extracts the topic from the title (`_topic_from_title` strips clickbait filler words — do NOT put the full title in the image prompt, the model renders it as garbled text; learned 2026-07), generates (a) a topic-literal dramatic background (subject right, no people) and (b) a separate **AI-generated reaction face** (`_thumbnail_reaction_prompt` — exaggerated shocked/amazed expression, built from the host's `AvatarSpec.description` with mood/expression phrases stripped out so they don't fight the reaction), both via the ImageProvider under one `plan.thumbnail` AssetRef (combined cost, one retry unit — thumbnails are cheap/non-critical). `render.render_thumbnail()` crops the background to 1280×720 `output/thumbnail.jpg` and composites the reaction face (`output/thumbnail_reaction.png`) as a white-ringed circular badge bottom-left; falls back to the plain `RENDERFLOW_AVATAR_IMAGE` portrait for older projects rendered before the reaction face existed, then to the first scene image if generation never ran. **Note: the reaction face is a fresh AI portrait matching the host's description, not a pixel-edit of the real avatar photo** — true identity-preserving editing (Pollinations' `kontext` image-to-image) needs the source photo hosted at a public URL, which a local file isn't; self-hosting one would be the path to a perfect identity match. The New Video modal has a "Make it clickbait" button that cycles the typed topic through title templates client-side, and a "🎲 Random topic" button (`RANDOM_TOPICS` bank, `web/index.html`) that filters out any title already matching an existing project (`state.projects`) before picking — never resuggests a story you've already made; shows an alert if every bank topic is used up.
- Dashboard render-state rule (`api.py`, `final_ready` in `_project_view`): a `final.mp4` counts as done only if **no run is currently active for that project, all assets are completed, and the file is newer than scenes.json** (rewritten on every asset update). The `not run` clause matters on its own: ffmpeg creates the output file on disk the instant it starts encoding (with a fresh mtime, before any content is valid), so without it a still-rendering video could briefly look "ready" — the client explicitly caught this ("don't show the download button while it's still rendering") before the `not run` gate was added.
- **`_file_url()` (api.py) appends `?v=<mtime>` to every asset URL** (scene thumbs, project thumbnail, final video). Scene/thumbnail/video filenames are deterministic (`scene_002.png`, `thumbnail.jpg`, `final.mp4`) — resuming or regenerating overwrites the same path, so without the cache-buster the URL is byte-identical to before and the browser keeps showing the stale cached image/video. Reported by the client as "regenerate scene isn't changing anything" — the backend was correct all along, it was purely a browser-caching symptom.
- **"Regenerate scene" also varies the prompt now** (`api.py::_vary_prompt`, used by the `/regenerate` endpoint only — not the automatic per-scene generation or the face-check retry, which vary by seed alone). A fresh random seed on the *exact same* prompt text often keeps a similar composition since the prompt drives composition far more than the seed does; a manual regenerate appends a randomly-chosen framing instruction (" Try this take: from a different angle.", etc.), stripping any previous variation clause first so repeated clicks don't grow the prompt unbounded. Skipped for solo-layout scenes (no background image to vary). Verified live: prompt updated, new file mtime, new cache-busted URL, all consistent.
- **Regenerate no longer auto-renders the final video** — `make_video.py --skip-render` runs every asset step (images/voice/avatar/subtitles/thumbnail) and stops before `render_video()`. `regenerate_scene` (api.py) always spawns with this flag: re-rendering a 50+ scene video after every single-scene fix used to hold the "run active" lock for minutes, during which no other scene could be touched — client-reported as "I can't regenerate another scene." After a skip-render run, the project correctly shows **Paused** (all assets complete, but `final.mp4` was deleted and not yet rebuilt) with a "Resume run" button to do the one render pass once the user is done fixing scenes. Verified live: a single-scene regenerate now takes ~19s (was minutes), and the subsequent render-only resume takes ~22s since every asset is skip-eligible.
- **Client-side unified action queue** (`web/index.html`: `state.actionQueues`, `drainActionQueues()`): only one `make_video.py` run may be active per project, so clicking "Regenerate" or changing a scene's avatar layout while a run is already active queues it instead of being blocked (regenerate button reads "Queue regenerate (run in progress)"; layout buttons show a ⏳ on the queued choice and stay clickable). Clicking a queued choice again cancels it; clicking a *different* queued layout option just changes what's queued in place. **One queue, not two** — `state.actionQueues[slug]` holds `{type:'regenerate', sceneId}` and `{type:'layout', sceneId, layout}` entries together, because two separate queues could both see `!project.running` in the same poll tick and fire simultaneously, racing each other against the backend's one-run-per-project check. `drainActionQueues()` (called every poll tick) fires at most one action per tick, in queued order. Purely client-side, no server-side queue — the backend still only ever runs one job per project at a time. Verified via state-machine simulation (mixed regenerate+layout queueing, cancellation, sequential draining) before trusting it live.
- **Auto-resume after the regenerate batch finishes** (`web/index.html`: `state.pendingAutoResume`, `maybeAutoResume()`): every `regenerateScene` call (immediate or queued) marks that project's slug pending; `maybeAutoResume()` runs at the end of every `drainRegenerateQueues()` tick and fires `/resume` automatically the moment the project is idle (`!running`) with an empty queue and status `Paused`/`Failed` — no manual "Resume run" click needed after fixing one or several scenes. The pending flag is cleared *before* firing (not after), so a status swing through Rendering → Complete on the next polls can never re-trigger it. Verified via state-machine simulation (single regenerate, and a multi-scene queued batch) before trusting it live: exactly one resume fires per batch, never mid-queue, never repeated after Complete.
  **If that automatic attempt fails** (non-OK response or a network error — e.g. a rare race where another run already started), it no longer fails silently: `notifyResumeNeeded(slug, title)` sets `state.resumeNeeded` and calls `playAlertSound()` (a two-tone chime synthesized via the Web Audio API — no external file, works fully offline, wrapped in try/catch since some browsers block audio before any user gesture on the page). `resumeAlertHtml()` renders a modal ("Action needed" — couldn't auto-start the render, click Resume) with **Resume now** (re-runs resume through the shared `post()`, which alerts normally if that *also* fails) and **Dismiss**. The success path stays silent by design — only the failure path is loud; client explicitly asked to keep automatic resume for the happy path and add notification only when it can't complete on its own.
  **Bug fixed 2026-07: `regenerateScene(id)` didn't convert `id` with `Number(...)`** — `data-arg` DOM attributes are always strings, but `sc.id` (used by every "is this scene queued" display check, and by the scene-grid queued badge) is a Number from the API JSON. `queue.includes(sc.id)` silently never matched, so the "Queued" state never rendered — the button looked completely inert. Worse, a second click on the same scene, thinking the first had no effect, would find the string in the queue via the *other* string-typed comparison inside the same action and silently cancel it. Client-reported as "the queue regenerate button is not working." Fix: `const id = Number(rawId)` as the first line of `regenerateScene`, matching the existing convention already used by `openScene`. Any future action that stores a scene id from `dataset.arg` for later comparison against `sc.id` must do the same.
- Run tracking survives API restarts: `_spawn` writes `logs/run.pid`; `_run_active` falls back to it and verifies the pid is a live `make_video.py` process. Assets orphaned in `running` state by a crashed/killed run are recovered on resume (`_start` routes running → failed → retrying → running). Learned the hard way: an in-memory-only `_runs` dict made live runs show "Paused" after a restart, and Resume then spawned a colliding duplicate that died with `InvalidTransition: running -> running`.
- **One failed scene no longer kills the batch.** `generate_images`/`generate_voice`/`generate_avatar_clips` (pipeline/assets.py) catch per-scene exceptions, log, mark that asset FAILED, and move on to the next scene instead of raising — a single stubborn provider error (e.g. a Pollinations rate limit) used to abort the entire run before any other scene was even attempted, catastrophic on a 50+ scene video. `make_video.py` now checks for incomplete scenes before rendering (`_incomplete_scenes`) and stops with a clear message ("scenes N, M still need assets — resume to retry") instead of an `AssertionError` mid-render. Learned the hard way 2026-07 on a 59-scene client-test video.
- **Captions**: `pipeline/subtitles.py` splits each scene's narration into ~4-word chunks and times them proportionally across the scene's *real* audio duration (no TTS provider gives word-level timestamps); each chunk renders to a transparent PNG (bold white text, black stroke, Arial Black/Bold from macOS system fonts) via Pillow. `assets.generate_subtitles()` runs after voice/avatar clips (needs their ffprobe'd duration) and is resume-aware via `Scene.assets.subtitle` (a plain `SubtitleRef`, no state-machine `.advance()` — status is set directly). `render.py` overlays the PNGs onto each scene clip with ffmpeg `overlay=...:enable='between(t,start,end)'` — works on both the zoompan and parallax visual paths and the avatar split-screen path. Pillow is now a **core** dependency (not an optional extra) since captions run by default on every video.
- **No scene-to-scene audio crossfade — a trailing pause instead.** A same-session attempt at `xfade`/`acrossfade` scene transitions was **removed**: `acrossfade` blends the tail of one clip's narration audio into the head of the next, so two different lines played on top of each other at every cut — reported live by the client-test viewer as "continuously speaking, previous line not even done yet." Fixed by baking a silent, still-frame pause into every scene clip instead of crossfading the final concat: `SCENE_GAP_SEC = 0.3` in `render.py`, applied per clip via `apad=pad_dur=...` on the audio and (for avatar clips, whose video ends exactly when their own narration does) `tpad=stop_mode=clone:stop_duration=...` to hold the last frame through the pause. `render_video()` is back to plain hard-cut `concat` — verified via RMS energy analysis on the real client video that each scene boundary is now a clean ~0.3-0.4s silence with no overlapping speech. **Lesson: audio crossfades are for music, not narration.** If transitions are wanted again, do a *video-only* dissolve during this same pause window — never touch the audio.
- **Avatar layout cycle, with a per-scene manual override**: solo full-screen vs. split-screen (avatar left + that scene's own visual right) defaults to a repeating cycle of 3 (`scene_is_avatar_solo()` in `pipeline/script.py`: `(scene_id - 1) % 3 == 0` makes scene ids 1, 4, 7, ... solo) — but the client asked for a way to force a specific scene either way when the generated visual doesn't fit ("sometimes user dont want the generated image... so he/she can change it to solo avatar talking instead"). `Scene.avatar_layout: "auto" | "solo" | "split"` (schema.py, defaults `"auto"` — old scenes.json files load fine with no migration) is a genuine per-scene override, persisted, checked first in `scene_is_avatar_solo(scene)` before falling back to the cycle. Every caller now takes the **whole `Scene`**, not `scene.id` — the function needs the override, and all 6 call sites already had the full scene in scope, so this was a clean signature change (`assets.py`, `render.py`, `api.py` ×3, `make_video.py`).
  Set via `POST /api/projects/{slug}/scenes/{scene_id}/layout` (`{"layout": "auto"|"solo"|"split"}`, `api.py::set_scene_layout`) — talking-avatar scenes only. Switching *to* split generates the missing background image first if one was never made (`--skip-render` spawn, same pattern as regenerate); switching *to* solo needs nothing generated and doesn't delete any existing image (harmless if they switch back later, no wasted regeneration). Either way `final.mp4` is deleted so the project shows **Paused**, picked up by the same auto-resume mechanism regenerate already uses. Dashboard: a 3-way segmented control in the scene side panel (`layoutSection` in `scenePanelHtml()`), plus the scene-grid card badge now reads "🧑 solo"/"🧑 split" (✦ marks a manual override) instead of a generic "🧑 host". Verified live end-to-end: forced a naturally-solo scene to split, watched it generate a real new background image, auto-render, and the final MP4 correctly show that scene split-screen.
  Solo scenes still skip generating an unused background image entirely (`assets.py::generate_images`, saves cost/time/rate-limit budget on ~1/3 of scenes by default) — every place that counts "is this scene's assets complete" (`api.py::_refs`, `api.py::_scene_assets`, `make_video.py::_incomplete_scenes`) had to be taught to not expect an `image` asset for solo scenes. `_scene_assets` (the per-scene UI chip list) was missed on the first pass and shipped a permanently-"pending" Image chip on every solo scene, reported by the client as "avatar scene remains avatar image pending"; fixed 2026-07, has a dedicated test (`tests/test_api.py`, the first test file for this module). `render.render_avatar_full_clip()` (full-screen) sits alongside the existing `render_avatar_split_clip()`; dispatch is in `render_scene_clip()`.
- **Dashboard "Overview" page** (`web/index.html`, nav item ◈): a presentation screen for client walkthroughs/Loom recordings — the pipeline stages, a free-stack-vs-paid-upgrade table per module (with $ figures), and an automated-vs-manual breakdown. Content is static (`PIPELINE_STAGES`/`STACK_MODULES`/`AUTOMATED_ITEMS`/`MANUAL_ITEMS` consts up top); only the cost tiles pull live numbers from `state.projects`. Keep this in sync by hand when the provider stack changes — nothing here is derived from `/api/state`.
- **`renderflow/api.py` does not hot-reload.** It's a long-running `uvicorn` process (unlike `make_video.py`, spawned fresh as a subprocess every run, which always reads current source) — edits to `api.py` need `pkill -f renderflow.api` + restart before they take effect. Learned the hard way 2026-07: the `final_ready` bugfix above and the avatar-layout dashboard changes silently didn't apply for a while because the server process predated the edits.
- **Dashboard logo**: `web/logo.png` (transparent PNG, red "Render Flow" wordmark) replaces the old red-square-+-text placeholder in the sidebar, served via a dedicated `GET /logo.png` route (`api.py`, same `FileResponse(WEB_DIR / ...)` pattern as `index()` — not folded into the `/files` static mount since that's scoped to the projects dir). Generated by cropping the tight bounding box out of the source `renderflow.png` (repo root, 1672×941 raw export) and keying near-black pixels to alpha so it sits cleanly on the dark sidebar background regardless of exact hex match.
- **Revenue/profit reporting** (manual entry — no YouTube API integration): `renderflow.schema.ProjectPerformance` (`views`, `watch_time_minutes`, `revenue_usd`, `notes`, plus `created_at`/`completed_at`/`updated_at`) is persisted per-project at `projects/<slug>/performance.json` (`storage.load_performance`/`save_performance`, gitignored like the rest of `projects/`). `created_at` is stamped automatically by `create_project` at project creation; `completed_at` is stamped automatically by `api._project_view` the first time `final_ready` goes true (both lazily backfilled on old projects that predate this feature, via `_best_effort_created_at` — `st_birthtime` where the platform has it, else the `scenes.json` mtime as an approximation). `_project_view` derives and returns `profit` (`revenue_usd - cost`, `None` until revenue is entered) and `productionTimeSec` (`completed_at - created_at`, `None` until the video finishes) on every poll — these are computed fields, never stored.
  `POST /api/projects/{slug}/performance` (`views`/`watchTimeMinutes`/`revenueUsd`/`notes`) is a **full replace of those four fields, not a partial merge** — the dashboard's Revenue tab always submits the whole form at once, so this is what lets clearing a field back to blank actually stick (a partial-merge design that skips `None` fields would make a value un-clearable once set). `created_at`/`completed_at` are untouched by this endpoint; only the pipeline sets those.
  Dashboard: a new **Reporting** nav page (`web/index.html`, nav item `$`) aggregates all projects — total cost/revenue/profit tiles, cost-by-module totals across every project, cost-per-video in creation order, and a per-video table (cost, revenue, profit, views, watch time, production time) sorted by profit descending; clicking a row jumps to that project's new **Revenue** tab (third tab alongside Pipeline/Cost) where the numbers are actually entered. Verified live against the real in-progress `the-solar-panel-trap` project (backfilled `created_at` correctly on an old project with no `performance.json`, and a manual save round-tripped through `/api/state` with `profit` computed correctly) and via a Node `vm`-sandboxed simulation of the new HTML-generating functions (profit sign/color, missing-data fallbacks, duration formatting) before trusting it live — same verification pattern used earlier in the session for the action-queue logic.

- **Currency normalization for speech** (`pipeline/text_normalize.py::normalize_for_speech`): Kokoro/Piper's espeak-based phonemizers read `$20,000` back digit-by-digit ("dollar two zero zero zero zero") instead of "twenty thousand dollars" — neither engine expands currency itself. A small dependency-free number-to-words converter expands `$`-prefixed comma-grouped amounts (with optional cents, e.g. `$20,000.50` → "twenty thousand dollars and fifty cents") and is applied in `assets.generate_voice()` **only to the string passed to `provider.synthesize()`**, never to `scene.narration` itself — captions (`pipeline/subtitles.py`) read the stored narration directly and should keep showing the numeral form on screen. Scoped narrowly to `$` currency patterns only (not general numbers/years) to avoid mangling things like dates. Not yet applied to ElevenLabs (cloud TTS generally normalizes currency itself) — revisit if that's shown to also mispronounce it.

Not built yet (roadmap order): stock-video B-roll (Pexels), subtitles (faster-whisper → ASS), narrator character consistency, more lip-sync avatar providers, background music/crossfades, Celery/Redis workers, Postgres, FastAPI, Next.js dashboard.

## Architecture — how a video gets made

```
make_video.py (CLI)
  [1] script:  topic --LLM--> GeneratedScript.to_plan() -> ScenePlan   (pipeline/script.py)
               or --scenes-file loads scenes.json directly (no LLM)
  [2] images:  per scene, ImageProvider.generate(image_prompt)          (pipeline/assets.py)
               talking_avatar scenes also get an avatar portrait
               (RENDERFLOW_AVATAR_IMAGE local file, or generated)
  [3] voice:   per scene, TTSProvider.synthesize(narration)             (pipeline/assets.py)
  [4] avatar:  talking_avatar scenes: AvatarProvider.generate_clip(
               portrait, voice, narration) -> lip-synced MP4            (pipeline/assets.py)
  [5] render:  per-scene clip (zoompan still, or avatar full/split),
               concat + loudnorm + faststart -> output/final.mp4        (pipeline/render.py)
```

Key mechanics that span multiple files:

- **`renderflow/schema.py` is THE central contract.** `ScenePlan` → `Scene` → `SceneAssets` (`image`, `avatar_image`, `voice`, `avatar_clip` as `AssetRef`s). Everything reads/writes these models; the plan is persisted to `projects/<slug>/script/scenes.json` after every asset state change. Treat schema changes like breaking API changes; update docs example and tests together. There is a parallel `Generated*` schema (what the LLM emits, no numeric constraints — unsupported by structured outputs) clamped into the real contract by `GeneratedScript.to_plan()`.
- **Asset state machine is exactly** `pending → running → completed | failed → retrying → running | cancelled`, enforced by `AssetRef.advance` (raises `InvalidTransition`).
- **Resume:** an asset with `status=completed` and a `path` is skipped on re-run. Consequence: config changes (voice, pacing, provider) do **not** retrofit an existing project — use a fresh `--slug` or delete the relevant `projects/<slug>/{voice,avatar}/` files.
- **Provider layer:** `providers/base.py` defines `Protocol`s (`LLMProvider`, `ImageProvider`, `TTSProvider`, `AvatarProvider`) returning `GeneratedAsset`/`LLMResult`; `providers/__init__.py` is the registry mapping env-selected names to adapters. `retry.py` supplies the backoff decorator adapters wrap their live calls with.
- **Rendering rules** (`pipeline/render.py`): scene duration always comes from ffprobe of the generated voice audio (or avatar clip) — never `duration_estimate_sec`. Scene visuals get **depth-parallax motion** by default (`pipeline/parallax.py`: Depth-Anything-V2-Small depth map + OpenCV remap, ~50 MB one-time HF download, ~2-5 s/scene on CPU; verified live 2026-07); `RENDERFLOW_MOTION=zoompan` opts out, and missing deps / depth failures fall back to zoompan automatically (flat Ken Burns, prescaled 2560×1440 to avoid sub-pixel jitter). Every talking-avatar scene renders split-screen (768px lip-synced avatar left, zoompan visual right); the local splitter marks **every** scene `talking_avatar`, so the host is on camera for the whole video (full-screen avatar path removed 2026-07). Audio is loudness-normalized per clip; final concat re-encodes with `+faststart`.
- **File formats ripple:** Piper emits WAV, ElevenLabs MP3 — `pipeline/assets.py` derives the file extension from `asset.meta["format"]`; lip-sync models want WAV, so `providers/avatar/postprocess.py` transcodes. That module also burns the "AI-generated host" disclosure banner into every avatar clip — **but the burn-in silently degrades to no banner when ffmpeg lacks `drawtext`, and the dev machine's ffmpeg build has no drawtext filter (discovered 2026-07), so local clips currently ship without the disclosure.** Fix path: reinstall ffmpeg with libfreetype, or overlay a pre-rendered PNG instead of drawtext.

## Layout

```
make_video.py               CLI entry point (the web API shells out to this too)
web/index.html              dashboard UI (vanilla JS, embedded fonts; polls /api/state
                            every 2.5s but re-renders only when state changed, preserving scroll)
renderflow/
  api.py                    FastAPI: /api/state, create/resume/regenerate/delete, /files static
  schema.py                 Scene JSON schema (Pydantic) + asset state machine — THE central contract
  config.py                 Settings from .env
  storage.py                projects/<slug>/{script,images,voice,avatar,subtitles,output,logs}/, plan save/load
  retry.py                  backoff decorator for external calls
```

## Hard rules

- **Never call an external API from business logic.** Everything goes through the `Protocol` interfaces in `providers/base.py` and the registry in `providers/__init__.py`. Fix response-shape problems in the provider adapter, not the pipeline.
- **Cost tracking from day one.** Every provider call records its cost on the asset/result; the system must always be able to answer "this video cost $X".
- **Never commit `.env`.** Keys live only there. `.env.example` documents the variables.
- Backend stays Python-only. Type hints everywhere; Pydantic for schema and API contracts.
- Keep changes minimal and surgical — Week 2 (workers, Postgres, FastAPI) will build on this structure; don't restructure preemptively.

## Provider notes (learned the hard way — do not rediscover)

- **ElevenLabs free tier blocks "library" voices over the API** (402 `paid_plan_required`). Only the account's *premade* voices work — e.g. George `JBFqnCBsd6RMkjVDRZzb` (documentary storyteller). Free tier = 10k chars/month, **no commercial rights**.
- **Pollinations** is keyless/free but anonymously caps at 1024×576 regardless of requested size (re-verified 2026-07); the renderer upscales, so output is 1080p but soft. A free registered token from auth.pollinations.ai (set `POLLINATIONS_TOKEN`) is sent as a Bearer header and lifts limits/watermark — whether it unlocks true 1080p is untested (no token yet). No `negative_prompt` input — the adapter folds it into the prompt. Can be slow (30–60 s/image). **Rate limits are real and sticky**: anonymous = 1 request/15 s, registered = 1/5 s; bursting past them earns persistent 429s that outlive short retries (hit 2026-07 after a 59-scene run). The adapter self-throttles to the allowed interval and retries with 20–60 s backoff — budget ~15-20 s/image anonymous.
- **Kokoro** (`kokoro-onnx`, Apache-2.0 — commercial OK) is the default narrator: far more natural prosody than Piper, runs local CPU faster than realtime, 24 kHz WAV. Setup: `pip install '.[kokoro]' && python scripts/setup_kokoro.py` (~340 MB into `.kokoro/`). Current voice `am_michael` (warm US male); alternates `bm_george` (older British), `am_onyx` (deep US), `am_fenrir`, `bm_fable`. `speed` = 1/`RENDERFLOW_TTS_LENGTH_SCALE` (mapped in make_video); sentence pauses inserted like Piper. No literal breath sounds — that tier is paid (ElevenLabs).
- **Piper** voices live in `.voices/` (~60 MB each). MIT license — commercial use OK. Downloaded: `en_US-john-medium` (deep older male), `en_US-lessac-medium`, `en_US-ryan-low`. Fast but robotic — replaced by Kokoro as default 2026-07. Client rule: visuals must be AI-generated, created for the story — **no stock photos/footage** (a Pexels provider was built and removed 2026-07).
- **dotenv gotcha:** `Settings.load()` uses `load_dotenv(override=True)` — `.env` is the source of truth even for pipeline subprocesses spawned by a long-running API server (without override, the server's stale inherited env pinned subprocess config; hit 2026-07 when a TTS switch didn't apply).
- **Piper pacing:** raw Piper sounds rushed. `RENDERFLOW_TTS_LENGTH_SCALE` slows delivery; the adapter also splits text on `.!?…` and inserts `RENDERFLOW_TTS_SENTENCE_PAUSE` seconds of silence between sentences (an ellipsis in narration earns a dramatic pause). Remember the resume gotcha: existing projects keep their old audio.
- **Wav2Lip avatar** (`wav2lip-local`): `scripts/setup_wav2lip.py` pulls code **and** weights from the `camenduru/Wav2Lip` HF mirror into `.wav2lip/` (~520 MB — the official OneDrive checkpoint links are dead) and patches `audio.py` for librosa ≥ 0.10 (kwargs-only `filters.mel`). Runs ~real-time on Apple-Silicon CPU (13 s narration → 15 s). Lips only — the model doesn't move the head, so the adapter adds a slow zoompan push-in; for real head motion use `sadtalker-replicate`. Verified live 2026-07.
- **SadTalker avatar** (`cjwbw/sadtalker` on Replicate, version pinned in the adapter): billed per GPU second, so cost scales with narration length (~$0.08–0.15 for a short clip; cost computed from `predict_time`). Defaults `preprocess=full`, `still_mode=False` (head motion), `use_enhancer=True`; if the background warps around the moving head, set `still_mode=True`. The account's Replicate token exists but is **unfunded** — a credit purchase (min $10) is required, a saved card is not enough (402 `Insufficient credit`).
- **MEMO avatar** (`memo-hf`, public `fffiloni/MEMO` Gradio Space via `gradio_client`): demo-grade only — the Space **trims input audio to 4 seconds** and requests a 240s ZeroGPU slot that anonymous users are refused outright; a free `HF_TOKEN` is required just to be admitted. Verified 2026-07: every public talking-head Space has similar guards (EchoMimic trims to 5s / requests 200–300s). **There is no free hosted lip-sync API fit for full-length scene narrations.** Also: `gradio_client>=2` renamed `hf_token=` to `token=`, and needs `httpx_kwargs={"timeout": httpx.Timeout(30, read=None)}` or long generations die with `ReadTimeout`.
- **Claude LLM step:** uses structured outputs (`output_config.format` with `GeneratedScript.model_json_schema()`); the generation schema deliberately has no numeric constraints (unsupported there) — bounds are clamped in `GeneratedScript.to_plan()`. No `ANTHROPIC_API_KEY` funded yet; the key-free workflow is: a Claude Code session writes `scenes.json` by hand → `--scenes-file`.

## Testing conventions

- Stub providers in `tests/stubs.py` satisfy the Protocols — use them; never hit live APIs in tests.
- The FFmpeg integration test generates media with lavfi and is skipped when ffmpeg is absent.
- Live provider verification is `scripts/check_providers.py`, run manually, never in CI. The `avatar` check is excluded from the default run (slow; costs money on paid providers).

## What NOT to do

- Don't generate AI video clips (only stills + motion) — cost control, Phase 4+ decision. For "alive" B-roll the planned route is free stock footage (Pexels API), not video generation.
- Don't build auth, billing, publishing, or dashboards before the pipeline they'd serve exists.
- Don't render synchronously inside API requests once the FastAPI layer exists (Week 2+) — queue it.
- Don't change the scene schema casually, and don't bypass the provider registry.
